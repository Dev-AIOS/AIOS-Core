# الجزء الخامس: تحسين التكاليف — كل تقنية معروفة لخفض تكلفة التوكنات

> **السياق الموحد**: منصة AI عربية (Aggregator/Reseller) تعيد بيع الوصول لنماذج AI بهامش ربح 25–40%.
> رأس المال شبه معدوم، VPS واحد، سعر الصرف: **1$ = 50 ج.م**.
> مزيج التوجيه المرجعي: مجاني → نماذج `:free`/رخيصة (~$0.05–0.10/1M)، مشترك → ~$0.30/1M، ثقيل → ~$0.50/1M ⚠️
> الاستهلاك المرجعي: مجاني ~200K توكن/شهر، أساسي ~1.5M، محترف ~5M.
> المعمارية المرجعية: LiteLLM proxy + Postgres + Redis + pgvector (من الجزء الرابع).

---

## 0. مقدمة: لماذا هذا الجزء هو "خزينة الربح"؟

في نموذج Reseller، المعادلة قاسية في بساطتها:

```
الربح = سعر البيع للمستخدم − (تكلفة التوكنات الفعلية + تكلفة البنية التحتية)
```

سعر البيع ثابت تقريباً (السوق يحدده، والمنافسة مع OpenAI/Anthropic مباشرةً مستحيلة على السعر).
المتغير الوحيد الذي نملكه بالكامل هو **المقام**: التكلفة الفعلية لكل توكن نمرره.

كل توكن نوفره — سواء منعنا إرساله أصلاً (Cache/Dedup)، أو ضغطناه (Compression)،
أو وجّهناه لنموذج أرخص (Routing)، أو حصلنا عليه بخصم من المزوّد (Prompt Caching / Batch) —
**يسقط مباشرةً وبكامله على خط الربح**.

### مثال رقمي محفّز (سنبنيه بالتفصيل في نهاية المستند)

| البند | قبل التحسين | بعد التحسين |
|---|---|---|
| تكلفة فعلية للمشترك المحترف (5M توكن/شهر @ ~$0.30/1M) | $1.50/شهر (75 ج.م) | ~$0.50/شهر (25 ج.م) |
| التكلفة الفعلية لكل 1M توكن | $0.30 | ~$0.10 |
| هامش الربح عند بيع 5M بـ $2.50 | 40% | **~80%** |

> ⚠️ **تنبيه منهجي**: النسب في هذا المستند تقديرات مبنية على أدبيات منشورة (GPTCache، LLMLingua،
> دراسات Anthropic/OpenAI الرسمية، تجارب فرق هندسية موثّقة) وعلى طبيعة حمل العمل العربي التفاعلي
> (chat عادي + أسئلة متكررة + RAG خفيف). أرقامك الحقيقية ستختلف؛ لذلك كل تقنية يجب أن تُقاس
> بمقياس A/B على بياناتك قبل تعميمها. القاعدة: **قِس أولاً، عمّم ثانياً**.

### كيف تقرأ هذا المستند؟

لكل تقنية ستة عناصر ثابتة:

1. **آلية العمل** — بعمق كافٍ لفهم *لماذا* توفر، لا فقط *كيف*.
2. **نسبة التوفير الواقعية** — مع مبرر الرقم، لا رقم تسويقي.
3. **متى تعمل ومتى تفشل** — حدود التطبيق.
4. **المخاطر والحدود** — بما فيها مخاطر الخصوصية والجودة.
5. **التنفيذ العملي في منصتنا** — مخطط/خوارزمية/أدوات مفتوحة المصدر فوق LiteLLM + Redis + Postgres/pgvector.
6. **أولوية التنفيذ** — P0 (الأسبوع الأول) / P1 (الشهر الأول) / P2 (الربع الأول).

## 1. Semantic Cache — التخزين المؤقت الدلالي

### 1.1 آلية العمل

الفكرة: كثير من الطلبات *مختلفة حرفياً* لكنها *متطابقة دلالياً*:

- «ما هي عاصمة فرنسا؟»
- «عاصمة فرنسا إيه؟»
- «قلي العاصمة بتاعة فرنسا»

كلها سؤال واحد. التطابق الحرفي (Exact-match) سيفشل، لكن **التطابق الدلالي** ينجح:

```
طلب جديد
   │
   ▼
[1] توليد Embedding للطلب (نموذج embeddings رخيص/محلي)
   │
   ▼
[2] بحث ANN في مخزن متجهات (pgvector / Redis vector / Qdrant)
    عن أقرب embedding مخزّن
   │
   ▼
[3] similarity ≥ عتبة θ (مثلاً 0.92 cosine)؟
   ├── نعم → أعِد الإجابة المخزنة — تكلفة التوليد = $0
   └── لا  → مرّر للنموذج، ثم خزّن (embedding, prompt, response, model, ts)
```

نقاط دقيقة في الآلية:

- **الـ embedding نفسه يكلف**: يجب استخدام نموذج embeddings رخيص جداً (مثل `text-embedding-3-small` بـ ~$0.02/1M) أو نموذج محلي مجاني (`bge-m3` أو `multilingual-e5-small` عبر `sentence-transformers` — ممتازة للعربية). تكلفة الـ embedding على سؤال من 50 توكن ≈ **$0.000001** — مهملة أمام توليد قد يكلف 500 ضعفها.
- **العتبة θ قرار اقتصادي لا تقني فقط**: عتبة منخفضة (0.85) = hit rate أعلى لكن خطر إجابات خاطئة؛ عتبة عالية (0.97) = أمان شبه كامل لكن hits أقل. نقطة البداية المعقولة: **0.92–0.95** للعربية مع `bge-m3`.
- **مفتاح الـ cache يجب أن يتضمن سياق النموذج**: إجابة من نموذج A لا تصلح بالضرورة لطلب موجّه لنموذج B مختلف القدرات (سنخزن `model_class` مع الإدخال).

### 1.2 نسبة التوفير الواقعية: **10–30%** من إجمالي التوكنات (قد تتجاوز 40% في منصات FAQ)

مبرر الرقم:

- ورقة GPTCache الأصلية وتجاربها المنشورة أظهرت hit rates بين 20% و70% حسب نوع حمل العمل؛ الحمل الأقرب لنا (أسئلة عامة متكررة من مستخدمين كثر) يقع في النطاق الأدنى-الأوسط.
- منصات chat عامة (وليست FAQ صرفة) لها تنوع دلالي أعلى، فينخفض الـ hit rate. تقدير متحفظ واقعي لحملنا المختلط: **15–25% hit**، وكل hit يوفر كامل تكلفة الطلب (input + output).
- التوفير الصافي = hit_rate × متوسط تكلفة الطلب − تكلفة embeddings + تخزين. بمتوسط طلب 800 توكن input و300 output عند $0.30/1M، وتكلفة embedding مهملة: hit واحد يوفر ≈ $0.00033.

### 1.3 متى تعمل ومتى تفشل

| تعمل بامتياز | تفشل / عديمة الجدوى |
|---|---|
| أسئلة عامة متكررة (FAQ، شرح مفاهيم، ترجمة جمل شائعة) | محادثات شخصية طويلة (كل رسالة تعتمد على سياق فريد) |
| مستخدمون كثر يسألون نفس الأسئلة (منصة عربية = قاعدة أسئلة مشتركة ضخمة) | طلبات توليد إبداعي («اكتب لي قصة عن...») — كل طلب فريد |
| طلبات RAG على مستندات ثابتة | أسئلة حساسة للوقت («أخبار اليوم»، «سعر الدولار الآن») |
| طبقة مجانية (Free tier) حيث الأسئلة سطحية ومتشابهة | طلبات بـ system prompt مخصص لكل مستخدم |

### 1.4 المخاطر والحدود — وعلى رأسها **تسريب البيانات بين المستخدمين** ⚠️

هذا هو الخطر الأكبر لهذه التقنية، ويجب التعامل معه كمسألة أمنية لا تحسينية:

**سيناريو الكارثة**: المستخدم أ يسأل «لخّص عقد العمل المرفق: يتقاضى محمد 45,000 ج.م شهرياً...».
النموذج يجيب. لو خزّنا هذا الزوج (سؤال/جواب) في cache عام، وسأل المستخدم ب سؤالاً دلالياً قريباً
(«كم راتب محمد في العقد؟»)، فسيحصل على إجابة مسرّبة من بيانات أ.

**استراتيجية العزل (إلزامية، غير قابلة للتفاوض)**:

```
cache_namespace = hash(plan_tier + scope + model_class)

scope ∈ { global_public, user:<id>, org:<id>, session:<id> }
```

1. **قاعدة ذهبية**: الـ cache العابر للمستخدمين (global) يُستخدم **فقط** للطلبات التي اجتازت
   فلتر «عمومية مؤكدة»: لا تحتوي مرفقات، لا تحتوي سياق محادثة سابق، وتطابق نية معروفة
   (سؤال عام، تعريف، ترجمة جملة قصيرة). كل ما عداه → cache خاص بالمستخدم فقط.
2. **Cache بمستويين**:
   - **L1 خاص**: لكل مستخدم (`user:<id>`) — آمن تماماً، يلتقط تكراراته الشخصية.
   - **L2 عام**: أسئلة عمومية مصنّفة فقط — يلتقط التكرار الجماعي.
3. **فلتر PII قبل الكتابة للـ L2**: كشف أرقام هواتف/بطاقات/إيميلات/أسماء بـ regex + NER خفيف؛
   أي طلب فيه PII لا يدخل L2 أبداً.
4. **TTL صارم**: L2 عمومي = 7 أيام افتراضياً (المعرفة تتقادم)، L1 خاص = 30 يوماً.
5. **قابلية الحذف**: زر «امسح بياناتي» يجب أن يمحي L1 الخاص بالمستخدم فوراً (التزام خصوصية أساسي).

مخاطر إضافية:

- **إجابات متقادمة**: سؤال «من رئيس وزراء بريطانيا؟» مخزن منذ سنة = إجابة خاطئة اليوم. الحل: TTL + قائمة نيات «حساسة زمنياً» تُستثنى من الـ cache (أي سؤال فيه «اليوم/الآن/أحدث/سعر»).
- **False positives دلالية**: «كيف أحذف حسابي؟» قد تطابق دلالياً «كيف أستعيد حسابي المحذوف؟» عند عتبة منخفضة. الحل: عتبة ≥0.92 + تخزين `intent tag` مع الإدخال ومطابقته أيضاً.
- **تضخم التخزين**: pgvector بملايين الإدخالات يبطئ. الحل: IVFFlat/HNSW index + حذف الإدخالات الأقل استخداماً (LRU eviction) عند تجاوز سقف الحجم.

### 1.5 التنفيذ العملي في منصتنا

عندنا خياران، والقرار: **نبدأ بـ pgvector (موجود أصلاً في معماريتنا) ونضيف Redis كطبقة L0 حرفية**:

```
LiteLLM proxy (custom pre-call hook)
   │
   ├──► L0: Exact-match في Redis         (تقنية 2 — أسرع من أي embedding)
   │         miss ▼
   ├──► L1: pgvector WHERE user_id = $1  (خاص، آمن)
   │         miss ▼
   ├──► L2: pgvector WHERE scope='global' AND intent ∈ whitelist
   │         miss ▼
   └──► النموذج الحقيقي → تخزين في L1 (+L2 إن اجتاز الفلاتر)
```

- **الأداة الأشهر**: [GPTCache](https://github.com/zilliztech/GPTCache) — يدعم Redis/Milvus/Postgres،
  وفيه `similarity_evaluation` قابلة للتخصيص. يمكن تشغيله كـ sidecar بجانب LiteLLM.
- **البديل الأبسط (موصى به لنا)**: custom middleware بـ ~150 سطر Python داخل LiteLLM proxy —
  لأن GPTCache يضيف تجريداً لا نحتاجه، ومنطق العزل (scope) عندنا حساس ونريد التحكم الكامل فيه.
- **نموذج الـ embeddings**: `intfloat/multilingual-e5-small` محلياً على الـ VPS (CPU كافٍ —
  embedding لجملة قصيرة يستغرق 10–30ms)، أو `text-embedding-3-small` سحابياً كخطة B.
- **قياس النجاح**: سجّل `cache_hit_rate` و`cache_false_positive_reports` في Postgres، وراجعهما أسبوعياً.

### 1.6 الأولوية: **P0** (مع فلتر العزل الأمني من اليوم الأول — لا cache بدون عزل)

## 2. Exact-match Cache — ذاكرة التطابق الحرفي

### 2.1 آلية العمل

أبسط تقنيات هذا المستند وأقدمها: خزّن `(key → response)` حيث:

```
key = SHA256( model + system_prompt + user_prompt + temperature + أي params مؤثرة )
```

قبل أي استدعاء، ابحث عن الـ key في Redis. موجود؟ أعد الرد. غير موجود؟ استدعِ وخزّن.

الفرق الجوهري عن الـ Semantic Cache: **صفر خطر دلالي** — لا يوجد false positive ممكن،
لأن التطابق حرفي 100%. إما نفس الطلب بالضبط أو miss.

### 2.2 نسبة التوفير الواقعية: **2–8%** إضافية فوق الـ Semantic Cache

مبرر الرقم:

- التطابق الحرفي الكامل نادر في chat حر (المستخدمون لا يكتبون نفس الجملة حرفياً غالباً).
- لكنه يلتقط حالات مهمة لا يلتقطها الدلالي بأمان: **إعادة إرسال نفس الطلب بعد انقطاع اتصال**،
  **ضغط المستخدم زر «إعادة التوليد»**، **عمليات batch متكررة** (نفس المستند يُلخّص مرتين)،
  **طلبات API آلية من تكاملات المستخدمين** (cron job يستدعي نفس البرومبت دورياً).
- دراسات caching التقليدية (خارج LLM) على أحمال API تظهر أن الطلبات المكررة حرفياً من مصادر آلية
  قد تشكل 5–15% من الحمل في منصات فيها مستخدمون-مطورون.
- ميزته الحاسمة: تكلفته شبه صفرية (Redis GET ≈ 0.1ms) وخطره صفر، فعائده/تكلفته ممتاز حتى عند 2%.

### 2.3 متى تعمل ومتى تفشل

| تعمل | تفشل |
|---|---|
| طلبات API آلية متكررة | chat بشري حر (نادراً ما يتكرر حرفياً) |
| أزرار «إعادة المحاولة» و«إعادة التوليد» | أي اختلاف ولو بمسافة واحدة = miss |
| وظائف batch دورية | طلبات فيها timestamps أو random seeds داخل البرومبت |

**نقطة خفية**: الـ key يجب أن يكون على **الـ payload الفعلي المرسل للمزوّد** بعد كل التحويلات
(بعد الضغط، بعد بناء السياق) — وإلا ستفوتك تطابقات أو تلتقط تطابقات زائفة عندما يتغير سياق المحادثة.

### 2.4 المخاطر والحدود

- **خطر تسريب البيانات نفسه كالتقنية 1** — نفس الحل: الـ key يتضمن `scope` (user id للطلبات الخاصة،
  `global` فقط للمصنّفة عمومية). الطلب الحرفي الواحد من مستخدمين مختلفين لهما بيانات مختلفة في السياق
  سيختلف keyهما تلقائياً إن أدخلنا سياق المحادثة في الـ hash — وهذا هو السلوك الصحيح أمنياً.
- **استهلاك ذاكرة Redis**: رد متوسط 1–2KB × ملايين المفاتيح. الحل: TTL (24–72 ساعة للعام) + `maxmemory-policy allkeys-lru`.
- لا يوجد خطر جودة إطلاقاً — الرد المخزن هو نفسه ما كان سيُولَّد (تقريباً؛ النماذج غير حتمية حتى عند temperature=0 أحياناً، لكن هذا فرق مهمل عملياً).

### 2.5 التنفيذ العملي

```
# داخل LiteLLM pre-call hook (pseudocode)
key = "exact:" + sha256(canonical_json({
    "model": model, "messages": messages,
    "temperature": temp, "scope": scope
}))
cached = redis.get(key)
if cached: return Response(content=cached, headers={"X-Cache": "HIT-EXACT"})
resp = call_llm(...)
redis.setex(key, ttl_by_scope(scope), resp.content)
```

- `canonical_json`: ترتيب المفاتيح ثابت، مسافات موحدة — وإلا فلن يتطابق أي شيء.
- سجّل في Postgres: `cache_layer ∈ {exact, semantic_l1, semantic_l2, miss}` لكل طلب — هذا الـ telemetry هو أساس كل قرارات التحسين اللاحقة.

### 2.6 الأولوية: **P0** — ساعات قليلة من العمل، خطر صفر، ويعمل كطبقة L0 أمام الـ Semantic Cache.

## 3. Prompt Compression — ضغط البرومبت (LLMLingua وما بعده)

### 3.1 آلية العمل

الفكرة: ليس كل توكن في الـ prompt يحمل معلومة. النماذج اللغوية نفسها تستطيع تقدير
أهمية كل توكن (perplexity-based). تقنية **LLMLingua** (Microsoft، 2023) تعمل هكذا:

```
Prompt طويل (مثلاً مستند RAG من 3000 توكن)
   │
   ▼
[1] نموذج لغوي صغير (LLaMA-7B / GPT-2-small) يحسب perplexity لكل توكن/عبارة
   │
   ▼
[2] التوكنات «قابلة للاستنتاج» (يسهل على النموذج تخمينها من سياقها) = حشو لغوي
    → تحذف أو تُستبدل
   │
   ▼
[3] الناتج: prompt مضغوط (قد يبدو غير طبيعي للبشر لكن النموذج الكبير «يفهمه»)
   │
   ▼
[4] يُرسل للنموذج الغالي → توفير = (التوكنات المحذوفة) × سعر الـ input
```

الجيل التالي من العائلة:

- **LongLLMLingua**: مصمم خصيصاً لسيناريوهات RAG الطويلة — يعيد ترتيب المستندات
  (الأهم أولاً، لأن النماذج تعاني Lost-in-the-Middle) ثم يضغط، مع الحفاظ على الإجابة عن السؤال.
- **LLMLingua-2**: ضغط كتصنيف token-level (احتفظ/احذف) عبر نموذج مدرّب — أسرع بكثير
  ونتائجه أفضل على النصوص متعددة اللغات.

### 3.2 نسبة التوفير الواقعية: **20–50% من توكنات الـ input في أحمال RAG/مستندات فقط** — أي ربما 5–15% من إجمالي فاتورة المنصة

مبرر الرقم:

- الورقة الأصلية تدّعي ضغطاً حتى ×20 مع فقدان أداء ضئيل على معايير محددة — لكن هذه أفضل حالاتها
  (استرجاع معلومة واحدة من مستند طويل جداً).
- النتائج الواقعية المستقرة عبر التجارب المستقلة: **×2 إلى ×4 ضغط** مع الحفاظ على >95% من الجودة،
  أي توفير 50–75% من طول المستند المضغوط.
- لكن انتبه: هذا ينطبق على **الجزء القابل للضغط من الحمل فقط** (مستندات RAG، نصوص طويلة لصقها المستخدم).
  رسائل chat القصيرة (غالبية حملنا) لا تُضغط عملياً — ضغط سؤال من 30 توكن يوفر 5 توكن ويكلف 200ms.
- اعتبار عربي مهم ⚠️: LLMLingua-2 دُرّب أساساً على الإنجليزية. على العربية، معدلات الضغط الآمن
  تنخفض (العربية أعلى كثافة معلوماتية لكل توكن في معظم الـ tokenizers). يجب قياس الجودة على
  عينة عربية قبل التفعيل، وتوقع ×1.5–×2 لا ×4.

### 3.3 متى تعمل ومتى تفشل

| تعمل | تفشل |
|---|---|
| مستندات RAG طويلة (>1000 توكن) | رسائل chat قصيرة (<200 توكن) — كلفة الضغط > عائده |
| نصوص لصقها المستخدم («لخّص هذا الكتاب») | برومبتات دقيقة بنيوياً (كود، JSON schema، تعليمات مرقّمة) — حذف توكن واحد يكسرها |
| سياقات قانونية/أكاديمية مطوّلة | المحتوى عالي الكثافة (قوائم أرقام، جداول، معادلات) |
| | العربية عند معدلات ضغط عالية (بدون معايرة مسبقة) |

### 3.4 المخاطر والحدود

- **فقدان معلومة حرجة**: الضغط تقديري؛ قد يحذف رقماً أو نفياً («لا يحق» → «يحق»). قاعدة صارمة:
  **لا ضغط أبداً** على: system prompts، تعليمات المستخدم المباشرة، الكود، البيانات المالية/الطبية،
  والأسئلة نفسها. الضغط فقط على **المستندات المرجعية المرفقة**.
- **التكلفة الحاسوبية للضغط نفسه**: تشغيل نموذج 7B للضغط على VPS بلا GPU = غير عملي (ثوانٍ لكل طلب).
  الحل: LLMLingua-2 بنموذج صغير (~0.5B) على CPU ≈ 100–300ms لمستند متوسط — مقبول لمستندات >1500 توكن فقط.
- **كسر الـ Prompt Caching الأصلي** (تقنية 12): البرومبت المضغوط يختلف في كل مرة ولو قليلاً،
  فيفشل التطابق مع prefix المخزن عند المزوّد. يجب ترتيب الطبقات: الضغط يحدث **قبل** بناء الـ final prompt،
  والمستند المضغوط يُخزَّن ويُعاد استخدامه (`doc_id → compressed_text`) حتى لا يُضغط نفس المستند مرتين.

### 3.5 التنفيذ العملي

```
في pipeline الـ RAG فقط (ليس على كل طلب):

if attached_doc_tokens > 1500:
    if cache.exists("compress:" + doc_hash):
        doc = cache.get(...)
    else:
        doc = llmlingua2.compress(doc, rate=0.5, force_tokens=["٪","$","0-9","لا","ليس"])
        cache.set("compress:" + doc_hash, doc, ttl=30d)
```

- الأداة: حزمة `llmlingua` الرسمية (open source, MIT) + نموذج `microsoft/llmlingua-2-xlm-roberta-large-meetingbank`
  — والـ xlm-roberta تحديداً لأنه **متعدد اللغات** (يدعم العربية بنسبة ما) بعكس البدائل الإنجليزية الصرفة.
- `force_tokens`: قائمة توكنات محمية من الحذف (أرقام، أدوات نفي، عملات) — تخفف أخطر فشلَين.
- معيار الإطلاق: اختبار على 200 مستند عربي → فقدان دقة الإجابة <2% قبل التفعيل، وإلا نؤجل.

### 3.6 الأولوية: **P1** — عائد حقيقي لكن محصور في أحمال المستندات، ويحتاج معايرة عربية قبل الثقة به.

## 4. Conversation Compression — ضغط/تلخيص سجل المحادثة القديم

### 4.1 آلية العمل

المحادثات الطويلة هي القاتل الصامت للميزانية. الرسالة رقم 40 في محادثة تُرسَل مع
**39 رسالة سابقة كـ input** — أي أن تكلفة الرسالة الواحدة تنمو خطياً مع طول المحادثة:

```
التكلفة التراكمية لمحادثة N رسالة ≈ O(N²) من حيث التوكنات المُعاد إرسالها
```

محادثة من 40 رسالة × 500 توكن/رسالة = **200K توكن input مُعاد إرسالها** عبر عمر المحادثة،
لو أرسلنا التاريخ كاملاً في كل مرة.

الحل: بمجرد تجاوز السجل عتبة (مثلاً 6000 توكن)، نلخّص الجزء القديم:

```
[ملخّص 150 توكن لأول 30 رسالة] + [آخر 10 رسائل كاملة] + [الرسالة الجديدة]
        بدلاً من
[40 رسالة كاملة = 20,000 توكن]
```

### 4.2 نسبة التوفير الواقعية: **60–90% من تكلفة المحادثات الطويلة**؛ على مستوى المنصة كاملة **10–25%**

مبرر الرقم:

- في المحادثة الملخّصة: 20,000 → ~2,150 توكن input = توفير ~89% لتلك المحادثة.
- لكن ما نسبة المحادثات التي تتجاوز العتبة؟ في منصات chat عامة، معظم الجلسات قصيرة
  (3–8 رسائل). المحادثات الطويلة (>20 رسالة) قد تكون 10–20% من الجلسات لكنها تستهلك
  40–60% من توكنات الـ input بسبب التراكم التربيعي. لذا التوفير الكلي 10–25% رقم واقعي.
- ملاحظة: هذا التوفير يتداخل بقوة مع تقنيتي 5 و6 — سنحسب التداخل في قسم «النموذج الحسابي».

### 4.3 متى تعمل ومتى تفشل

| تعمل | تفشل |
|---|---|
| محادثات دعم/تعلم طويلة (سياق تراكمي لكن التفاصيل القديمة غير حرجة) | تحرير مستند واحد عبر عشرات التعديلات — التفاصيل القديمة كلها حرجة |
| جلسات «عصف + تنفيذ» (المراحل القديمة تُلخَّص بأمان) | محادثات فيها تعليمات قديمة ما زالت سارية («تذكّر: لا تستخدم أبداً X») |
| | محادثات قانونية/طبية حيث كل كلمة سابقة قد تُستدعى |

### 4.4 المخاطر والحدود

- **فقدان تعليمات قديمة سارية**: المستخدم قال في الرسالة 3 «أجب دائماً بالمصري»، الملخّص أسقطها،
  فانزعج المستخدم عند الرسالة 35. الحل: **تعليمات المستخدم الصريحة تُستخرج وتُخزَّن منفصلة**
  (في `user_preferences` بجانب الـ system prompt) ولا تمر بالملخّص أصلاً.
- **تكلفة التلخيص نفسه**: التلخيص استدعاء LLM إضافي! قاعدة حسابية:
  التلخيص مربح فقط إذا `تكلفة التلخيص < التوفير المتوقع من الضغط × عدد الرسائل المستقبلية المتوقعة`.
  عملياً: لخّص بنموذج رخيص ($0.05–0.10/1M) — تلخيص 6000 توكن ≈ $0.0006، ويوفر ~$0.005 في كل رسالة لاحقة عند $0.30/1M.
- **التلخيص المتكرر (summary-of-summary) يتحلل**: بعد 3–4 دورات تلخيص يصبح الملخّص ضبابياً.
  الحل: احتفظ بـ `summary_version`، وبعد 4 دورات أبقِ «الحقائق الثابتة المستخرجة» فقط
  (أسماء، قرارات، أرقام) بدل ملخّص نثري.
- **كسر Prompt Caching** (تقنية 12): تغيير التاريخ في كل رسالة يكسر الـ prefix cache عند المزوّد
  جزئياً — لكن الـ system prompt الثابت يبقى قابلاً للتخزين. صمّم الترتيب: `[system ثابت][ملخّص][تاريخ حديث]`
  بحيث يكون الجزء المتغير في النهاية قدر الإمكان (تفاصيل في تقنية 12).

### 4.5 التنفيذ العملي

```
# Postgres: جدول conversations
(id, user_id, full_history JSONB,           # الأرشيف الكامل — لا يُرسل للنموذج
 summary TEXT, summary_upto_msg INT,         # الملخّص وحدوده
 prefs JSONB)                                # تعليمات مستخرجة

# خوارزمية بناء الـ prompt لكل رسالة جديدة:
tokens_recent = count(آخر K رسائل)
if tokens_recent + summary_tokens > budget_for_tier(plan):
    # لخّص الرسائل من summary_upto_msg حتى (N - K)
    new_summary = cheap_model.summarize(old_summary + الرسائل_الوسطى)
    update row
final_prompt = [system] + [prefs] + [summary] + [آخر K رسائل] + [الرسالة الجديدة]
```

- K (عدد الرسائل الكاملة المحفوظة) يتكيف مع الخطة: مجاني K=4، أساسي K=8، محترف K=15
  (انظر تقنية 18 — Adaptive Context).
- استخدم النموذج الرخيص نفسه المستخدم في طبقة التوجيه المجانية للتلخيص — لا تدفع $0.30/1M لتلخيص ستوفر به $0.30/1M.

### 4.6 الأولوية: **P0** — أعلى عائد على المحادثات الطويلة، وتنفيذه مباشر فوق Postgres الموجود.

## 5. Context Optimization — اختيار ما يدخل نافذة السياق فعلاً

### 5.1 آلية العمل

السؤال التأسيسي قبل أي طلب: **ما هي أصغر مجموعة معلومات تجيب هذا السؤال؟**
في معظم المنصات، الـ prompt يُبنى بعقلية «ألقِ كل شيء في النافذة». التحسين يعني بوابة اختيار:

```
مصادر السياق المحتملة:
  [system prompt] [ذاكرة المستخدم] [تاريخ المحادثة] [مستندات RAG] [نتائج أدوات] [السؤال]
        │
        ▼
  بوابة الاختيار (per-source budget + relevance gating)
        │
        ▼
  [فقط ما يلزم] ──► النموذج
```

مبادئ الاختيار:

1. **RAG retrieval بحد أقصى ذكي**: لا `top-k=10` ثابتة. استخدم `top-k` متكيفاً مع عتبة صلة:
   أدخل فقط المقاطع التي similarity ≥ 0.75 (مثلاً)، وبحد أقصى 4 مقاطع. كثير من الطلبات تجيبها مقطعة واحدة.
2. **قصّ المقاطع المسترجعة**: استرجع 300 توكن من المستند لا 3000 — chunk صغير حول موضع التطابق.
3. **استبعاد مصادر كاملة عند عدم الصلة**: سؤال «اشرح لي الـ recursion» لا يحتاج ذاكرة المستخدم
   ولا مستندات — نوايا كثيرة تحتاج system + سؤال فقط. صنّف النية (نفس classifier التوجيه، تقنية 10)
   واربط لكل نية قائمة مصادر مسموحة.
4. **إزالة التكرار داخل السياق**: مقطعا RAG متشابهان 90%؟ أبقِ واحداً (dedup داخلي — امتداد لتقنية 8).

### 5.2 نسبة التوفير الواقعية: **15–35% من توكنات الـ input الكلية**

مبرر الرقم:

- دراسات Lost-in-the-Middle (Liu et al., 2023) أظهرت أن النماذج تتدهور مع السياقات المحشوة —
  أي أن الحشو الزائد لا يكلف مالاً فقط بل **يخفض الجودة**. القصّ ربح مزدوج.
- في أحمال RAG النمطية، 60–80% من المقاطع المسترجعة بـ top-k ثابتة تكون غير مستخدمة فعلياً
  في الإجابة النهائية (يمكن التحقق بالـ citation analysis).
- لتطبيقنا: المنصة عربية chat-محورية مع RAG خفيف، فالتقدير المتحفظ 15–35% من الـ input،
  والـ input عادة 60–75% من الفاتورة → **10–25% من الفاتورة الكلية**.

### 5.3 متى تعمل ومتى تفشل

| تعمل | تفشل |
|---|---|
| أي نظام RAG بـ top-k ثابتة (دائماً فيه هدر) | سياقات قصيرة أصلاً (لا شيء لقصّه) |
| نيات واضحة المعالم (أسئلة معرفية، أكواد، ترجمة) | مهام توليفية تحتاج السياق كله («قارن كل الفصول») |
| مستخدمون يلصقون مستندات ضخمة لسؤال صغير | عندما تكون عتبة الصلة مضبوطة خطأً فتُسقِط المقطع الحاوي للإجابة |

### 5.4 المخاطر والحدود

- **قصّ المقطع الخطأ** = إجابة خاطئة واثقة. هذا أخطر من الهدر. الحل: عتبة صلة متحفظة أولاً،
  ثم إحكامها تدريجياً مع مراقبة معدل «الإجابة لم تجب السؤال» (إشارات: إعادة صياغة المستخدم لسؤاله،
  تقييمات سلبية).
- **تعقيد الـ pipeline**: كل بوابة اختيار نقطة فشل جديدة. ابدأ بقاعدتين بسيطتين (RAG gating +
  استبعاد الذاكرة للنوايا المعرفية) وتوسّع لاحقاً.
- يتداخل مع تقنية 4 (كلاهما يقلص input المحادثات) — لا تضرب نسبتيهما في بعضهما.

### 5.5 التنفيذ العملي

```python
# داخل الـ middleware قبل بناء الـ prompt النهائي
SOURCE_MATRIX = {
    "factual_qa":   ["system", "question"],                    # لا تاريخ، لا ذاكرة
    "code_help":    ["system", "question", "recent_2_msgs"],
    "doc_qa":       ["system", "question", "rag_top4_gated"],
    "personal_chat":["system", "summary", "recent_K", "prefs"],
    ...
}
# RAG gating
chunks = vector_search(q, top_k=8)
chunks = [c for c in chunks if c.score >= 0.75][:4]
chunks = dedupe_chunks(chunks, sim_threshold=0.9)
chunks = [truncate_around_match(c, window=300) for c in chunks]
```

- كل قرار اختيار يُسجَّل (`sources_used`, `tokens_before_gate`, `tokens_after_gate`)
  لقياس التوفير الفعلي لكل قاعدة.
- **P0 vs P1**: القاعدتان البسيطتان (gating + source matrix) = **P0**؛ القصّ الذكي حول موضع
  التطابق والـ dedup الداخلي = **P1**.

### 5.6 الأولوية: **P0** (القواعد الأساسية) / **P1** (التفاصيل المتقدمة)

## 6. Memory Summarization — ذاكرة ملخّصة بدل إعادة إرسال التاريخ

### 6.1 آلية العمل

تقنية 4 تضغط *داخل* المحادثة الواحدة. هذه التقنية تعمل *عبر* المحادثات:
بدل أن يبدأ كل chat جديد من الصفر (فيفقد المستخدم تفضيلاته) أو من إعادة إرسال أرشيف
محادثات قديمة (كارثة تكلفة)، نبني **ذاكرة مستخدم مدمجة** تُحدَّث دورياً:

```
نهاية كل محادثة (أو كل 10 رسائل):
   │
   ▼
نموذج رخيص يستخرج: [حقائق عن المستخدم] [تفضيلات] [قرارات سابقة] [مواضيع جارية]
   │
   ▼
تُدمج في memory card (حد أقصى 300–500 توكن)
   │
   ▼
كل محادثة جديدة تبدأ بـ: [system] + [memory card] + ...
   بدل: [system] + [أرشيف 50,000 توكن]
```

الفرق الدقيق عن تقنية 4: تلك *ضغط سجل حي*، وهذه *استخلاص معرفة طويلة الأمد* —
memory card تُكتب مرة وتُقرأ آلاف المرات بتكلفة ثابتة صغيرة.

### 6.2 نسبة التوفير الواقعية: **منع كارثة أكثر منه توفير مباشر — 5–15% مع تجنّب نموّ غير محكوم**

مبرر الرقم:

- بدون ذاكرة ملخّصة، البديلان كلاهما سيئ: (أ) إرسال الأرشيف = تكلفة تنفجر مع عمر الحساب،
  (ب) بلا ذاكرة = تجربة رديئة ترفض churn. التوفير الحقيقي = الفرق بين (أ) والذاكرة الملخّصة:
  مستخدم سنة معنا قد يراكم 500K توكن أرشيف؛ الذاكرة الملخّصة 400 توكن.
- لأغلبية المستخدمين الجدد (أرشيف صغير)، الأثر المباشر متواضع: 5–15% على مستوى المنصة.
- الفائدة الثانية غير محسوبة في النسبة: الذاكرة الملخّصة ترفع الجودة المدركة (تخصيص) فتخفض churn —
  وهذا FinOps أيضاً.

### 6.3 متى تعمل ومتى تفشل

| تعمل | تفشل |
|---|---|
| مستخدمون متكررون بعلاقة طويلة مع المنصة | مستخدم عابر بجلسة واحدة (لا شيء لاستخلاصه) |
| تفضيلات مستقرة (أسلوب، لغة، مجال عمل) | سياقات متناقضة عبر الزمن («انسَ كل ما قلته سابقاً») |
| | مهام تتطلب تفاصيل حرفية قديمة («ما الرقم الذي ذكرته في مارس؟») |

### 6.4 المخاطر والحدود

- **ذاكرة خاطئة عنيدة**: استخلاص خاطئ («المستخدم طبيب» وهو صيدلي) سيلوّث كل محادثة لاحقة.
  الحل: اجعل الذاكرة قابلة للاطلاع والتعديل من المستخدم («هذا ما أعرفه عنك») — وهذا التزام
  شفافية وخصوصية أيضاً (GDPR-روح).
- **نموّ الذاكرة نفسها**: memory card بلا حد تصبح أرشيفاً جديداً. سقف صارم (500 توكن) + دمج
  دوري يحلّ التعارضات (الأحدث يغلب).
- **خصوصية**: الذاكرة بيانات شخصية مركّزة. تشفير at-rest + حذف فوري عند طلب المستخدم +
  عدم إدخالها أبداً في أي cache مشترك (تقنية 1).
- يجب ألا تُرسل الذاكرة كاملة في كل رسالة: اربط إرسالها بالنية (تقنية 5 — `SOURCE_MATRIX`).

### 6.5 التنفيذ العملي

```python
# job دوري (cron أو بعد إغلاق محادثة > K رسائل)
facts = cheap_model.extract(convo, schema={
    "preferences": [...], "facts": [...], "open_threads": [...]
})
memory = merge(old_memory, facts, max_tokens=500)  # دمج مع حلّ تعارض
db.upsert_user_memory(user_id, memory)
```

- الاستخلاص بنموذج رخيص وبتعليمات JSON صارمة (تقنية 11).
- الدمج قاعدة + LLM: الحقائق المؤرخة (key-value) تُدمج بقواعد، والنصوص الحرة تُدمج بتلخيص.
- قابل للتنفيذ كاملاً فوق Postgres الموجود (`user_memory` table + JSONB).

### 6.6 الأولوية: **P1** — ليست أول ما يُبنى (المستخدمون الجدد لا يستفيدون)، لكنها ضرورية قبل أن يتراكم الأرشيف.

## 7. Shared Cache عبر المستخدمين — ذاكرة الأسئلة الشائعة العامة

### 7.1 آلية العمل

هذا هو المستوى L2 من تقنية 1، لكنه يستحق فصلاً مستقلاً لأنه قرار **منتج وأخلاق وقانون**
قبل أن يكون قراراً تقنياً. الآلية: طلب مصنّف «عمومي خالص» (تعريف مصطلح، شرح مفهوم، ترجمة
جملة عامة، سؤال برمجي شائع) يُبحث له في cache مشترك بين كل المستخدمين:

```
مستخدم جديد يسأل «ما هو الـ API؟»
   │
   ▼
[مصنّف عمومية: عام ✓، لا مرفقات ✓، لا سياق شخصي ✓، لا PII ✓، نية whitelisted ✓]
   │
   ▼
بحث دلالي في shared_faq_cache
   ├── hit → إجابة فورية، تكلفة ≈ $0، زمن استجابة <100ms
   └── miss → النموذج → الرد يُضاف للـ shared cache بعد اجتياز الفلاتر
```

### 7.2 نسبة التوفير الواقعية: **5–20% إضافية، وتتركز في الطبقة المجانية**

مبرر الرقم:

- في منصة عربية، الأسئلة المعرفية العامة متشابهة جداً عبر المستخدمين (تعليم، دين، صحة عامة،
  برمجة أساسية، ترجمة). منحنى Zipf حاد: أشهر 1000 سؤال قد تغطي 15–30% من طلبات الطبقة المجانية.
- لكن الطبقة المجانية أصلاً موجّهة لنماذج شبه مجانية ($0.05–0.10/1M)، فالتوفير المطلق بالدولار
  صغير — قيمته الحقيقية: **سرعة استجابة فورية** (تجربة أفضل) و**تخفيف الحمل عن حدود معدل
  النماذج المجانية** (rate limits هي عنق الزجاجة الحقيقي للطبقة المجانية).
- في الطبقات المدفوعة، الأسئلة أخصّ وأكثر تنوعاً → hit rate أدنى بكثير (3–8%).

### 7.3 هل هو قانوني وأخلاقي؟ — النقاش الصريح

**الحجج الداعمة (لماذا هو مقبول بشروط):**
- سؤال «ما هي عاصمة اليابان؟» وإجابته ليسا ملكاً لأحد؛ إعادة استخدام معرفة عامة لا تسرّب شيئاً.
- يشبه مبدأ CDN caching وWikipedia — تجميع المعرفة العامة وتقديمها بكفاءة.
- بيئياً وحوسبياً: منع إعادة توليد نفس المعرفة ملايين المرات كفاءة مشروعة.

**الحجج الرادعة (وحدود المشروعية):**
1. **تسريب بيانات**: أخطر فشل — إجابة تحوي معلومة من سياق مستخدم أعيدت لمستخدم آخر.
   هذا خرق خصوصية صريح وقد يخالف قوانين حماية البيانات (مصر: قانون 151/2020؛ وأيضاً GDPR
   لو كان عندك مستخدمون أوروبيون). → الحل: فلتر عمومية صارم، والقاعدة «عند الشك: خاص».
2. **شروط المزوّدين**: بعض شروط استخدام API للمزوّدين تقيّد تخزين/إعادة استخدام المخرجات
   (تاريخياً: OpenAI تسمح بالتخزين وتمليك المخرجات للمستخدم؛ بعض المزوّدين يمنعون استخدام
   المخرجات لتدريب نماذج منافسة — caching للخدمة لا للتدريب عادة مقبول، لكن **يجب مراجعة
   ToS لكل مزوّد نستخدمه وتوثيق المراجعة**). هذه خطوة إلزامية قبل الإطلاق، ونقطة تدقيق قانونية حقيقية.
3. **التناسب**: طلب مستخدم مدفوع أُجيب من cache عام بدل نموذجه المميز — هل حصل على ما دفع ثمنه؟
   → قاعدة العدل: الـ shared cache يخدم الطبقة المجانية والأسئلة العامة فقط؛ المشترك المدفوع
   الذي يتوقع نموذجاً قوياً يحصل عليه (أو يُخبَر شفافاً).
4. **الإسناد والجودة**: إجابة مخزنة قديمة قد تكون خاطئة اليوم (راجع مخاطر التقنية 1 — TTL
   وقوائم الاستثناء الزمني).

**الخلاصة الأخلاقية العملية**: مقبول ومبرر بشروط صارمة — whitelist نيات، فلتر PII، TTL قصير
للموضوعات المتغيرة، شفافية في سياسة الخصوصية («قد نستخدم إجابات محفوظة للأسئلة العامة»)،
ومراجعة ToS المزوّدين. وأي شك = لا يُخزَّن مشتركاً.

### 7.4 التنفيذ العملي

```python
def eligible_for_shared_cache(request, intent):
    return (intent in {"definition", "general_concept", "translation_short",
                       "code_snippet_common", "howto_generic"}
        and not request.has_attachments
        and not request.has_conversation_history
        and not pii_detected(request.text)
        and not temporal_sensitive(request.text)   # "اليوم"، "سعر"، "آخر"
        and len(request.text) < 500)
```

- جدول `shared_cache` في Postgres + pgvector، منفصل تماماً عن caches الخاصة.
- لوحة مراجعة داخلية: عينة أسبوعية عشوائية من مدخلات الـ shared cache تُراجَع يدوياً بحثاً عن
  تسرّب (audit) — في الأشهر الأولى على الأقل.
